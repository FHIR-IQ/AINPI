# 2026-06-09 update — can you associate a practitioner with a phone number?

Short answer for anyone who asked (hi Ron): **yes — and the *where* turned
out to be the surprise.** We pre-registered a new metric, **H43 — practitioner
phone-number reachability**, expecting the federal directory to hide phone
numbers one or two hops away from the practitioner. We ran it against the live
warehouse. The data said the opposite. This is pre-registration working as
intended: the hypothesis was wrong, and the measurement corrected it.

## The question, and the assumption we brought to it

In FHIR R4 / the NDH Implementation Guide a practitioner's phone can sit in
three places:

1. **`Practitioner.telecom`** — on the individual's own record.
2. **`PractitionerRole.telecom`** — on the role tying the practitioner to an org.
3. **`Location.telecom`** — the practice location, via `PractitionerRole.location → Location`.

The conventional wisdom — because NPPES, the upstream of ~90% of these fields,
keeps practice phone on the *location* — is that `Practitioner.telecom` is
sparse and you have to traverse role → location to actually find a phone. We
pre-registered H43 around exactly that expectation.

## What the 2026-05-08 release actually shows

| Metric | Value |
|---|---|
| Active Practitioner resources (denominator) | **7,196,385** |
| Phone **directly on** `Practitioner.telecom` | **7,195,270 (99.98%)** |
| Reachable *only* via role / location traversal | **0** |
| No phone on **any** of the three resources | **1,115 (0.015%)** |
| On-record fax | 2,871,690 (~40%) |
| On-record email / url | 0 / 0 |

**The phone is right on the Practitioner record — for essentially everyone.**
NDH evidently enriches `Practitioner.telecom` directly rather than leaving the
number stranded on the location. The PractitionerRole → Location traversal we
built to "rescue" reachability adds nothing here, and only about a thousand
active practitioners in the entire 7.2-million-row population have no phone on
any of the three resources.

So the operational takeaway flips: a "call this provider" feature **can** read
`Practitioner.telecom` straight off the Practitioner. You do not need the
role/location hop for phone in this release — keep it only as a fallback for the
~1,115 records with nothing on file.

## Why we ran it instead of shipping the assumption

The first instinct was to write up "phone lives on the location, traverse to
find it" as settled fact. It reads plausibly and it matches how raw NPPES
behaves. It is also wrong for the NDH bulk export. The only reason this update
states the true result is that the metric is computed by a committed,
reproducible script —
[`analysis/h43_practitioner_phone.py`](https://github.com/FHIR-IQ/AINPI/blob/main/analysis/h43_practitioner_phone.py)
— run against BigQuery with the project's 100 GB per-query cap, with the output
JSON published at
[`/api/v1/findings/practitioner-phone-reachability.json`](https://ainpi.dev/api/v1/findings/practitioner-phone-reachability.json).
Numbers first, narrative second.

## Caveats worth carrying forward

- **Release-sensitive location layer.** The 2026-05-08 release deduped
  `Location` resources by 61% versus April. Phone reachability does not depend
  on that layer in this release (it is ~100% on-record), but path-3 numbers will
  move release to release.
- **Empty channels.** On-record telecom in this release is phone + fax only —
  `email` and `url` came back at zero, and the location-phone path came back
  empty. Those zeros don't affect the headline (on-record phone stands on its
  own), but confirm against the source before relying on email/url/location
  contact channels.

## The takeaway

Can you associate practitioners with phone numbers in NDH? Yes — and more
directly than the FHIR plumbing suggests. 99.98% carry a phone right on the
Practitioner record. The pre-registered expectation that you'd have to chase it
through role and location did not survive contact with the data.
