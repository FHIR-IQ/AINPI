# 2026-06-09 update — can you associate a practitioner with a phone number?

Short answer for anyone who asked (hi Ron): **yes — but "the phone number"
is not one field, and where it lives is the actual finding.** This update
pre-registers a new metric, **H43 — practitioner phone-number reachability**,
and ships the compute script. As with the 2026-06-02 landscape release, the
live cell numbers land on the next weekly-refresh; the methodology, the
script, and the pre-registration record are public now.

## The question is sharper than it looks

In FHIR R4 and the NDH Implementation Guide, a practitioner's phone can sit in
three different resources:

1. **`Practitioner.telecom`** — on the individual's own record.
2. **`PractitionerRole.telecom`** — on the role that ties the practitioner to
   an organization.
3. **`Location.telecom`** — on the place the practitioner practices, reached
   by following `PractitionerRole.location → Location`.

This matters because **NPPES — the upstream source of roughly 90% of these
fields — keeps practice phone on the location, not the individual.** So if you
read `Practitioner.telecom` and find it empty, that is not (usually) a data
quality defect. It means the phone is one or two hops away, on the role or the
location.

If you build a "call this provider" feature against `Practitioner.telecom`
alone, it will look like the federal directory has almost no phone numbers.
That conclusion would be wrong — and H43 exists to measure exactly how wrong.

## What H43 measures

- **Denominator:** active `Practitioner` resources in the pinned NDH release
  (~7.44M at 2026-05-08).
- **Numerator:** distinct active practitioners reachable by a `system='phone'`
  telecom entry via **any** of the three paths — unioned, then intersected
  back to the active set so the numerator can never exceed the denominator and
  dangling references drop out.
- **The headline split:** *phone on the Practitioner record* vs *phone only
  reachable by traversing PractitionerRole → Location*. That split is the real
  answer — it tells an integrator which resource to read.

The result renders as three bars that sum to the full practitioner population:

- **Phone on Practitioner record** — published directly on the individual.
- **Phone only via role / location** — discoverable, but only by traversal.
- **No phone on any path** — not reachable through any of the three resources
  in NDH (for these, fall back to the NPPES practice-location phone).

## Why "pre-registered" and not a number tonight

The aggregation script — [`analysis/h43_practitioner_phone.py`](https://github.com/FHIR-IQ/AINPI/blob/main/analysis/h43_practitioner_phone.py)
— is committed and runs against BigQuery with the project's 100 GB per-query
cap (`bq_job_config()`). It is wired into the weekly-refresh cron, which
regenerates `/api/v1/findings/practitioner-phone-reachability.json` and commits
it to `main`. Until that run lands, the
[finding page](https://ainpi.dev/findings/practitioner-phone-reachability)
shows the pre-registration record — null hypothesis, denominator, data source,
method — with no fabricated numbers. This is the same register-before-numbers
discipline every AINPI finding follows.

## One load-bearing caveat

The 2026-05-08 release **deduped `Location` resources by 61%** versus April.
Because path 3 (phone via the referenced location) depends on the Location
table, that dedup mechanically lowers location-path reachability across the
release boundary. It is a change in how CMS packaged the data, not a loss of
contact information — but any cross-release comparison of H43 has to hold it
constant. The finding notes say so on the page, and the provenance doc
([`docs/methodology/runs/2026-06-09-h43-practitioner-phone.md`](https://github.com/FHIR-IQ/AINPI/blob/main/docs/methodology/runs/2026-06-09-h43-practitioner-phone.md))
records the method an auditor can reproduce.

## The takeaway

Can you associate practitioners with phone numbers in NDH? Yes — by reading
the right resource. The phone is rarely on `Practitioner.telecom`; it is
overwhelmingly on the **role** and the **location**. H43 turns that from a
folk-knowledge gotcha into a measured, reproducible number you can cite.
