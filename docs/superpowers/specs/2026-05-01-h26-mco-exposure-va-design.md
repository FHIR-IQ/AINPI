# H26 — MCO networks containing federally excluded providers (Virginia pilot)

**Status:** design approved, pending implementation plan
**Date:** 2026-05-01
**Author:** spec generated via brainstorming with Eugene Vestel
**Methodology version target:** 0.5.0

## Goal

Cross-reference the AINPI high-risk cohort (federally excluded subset)
against live FHIR provider directories of payer entities reachable via
their public Da Vinci PDex Plan-Net endpoints, and publish per-payer
match counts as a citable finding. Anchored in 42 CFR § 455.436
(federal database checks) and § 438.602 (Medicaid managed care
directory oversight).

**v1 endpoint set (4):** Anthem HealthKeepers Plus (the VA Medicaid
brand), Anthem Blue Cross (broader Anthem network), Humana, and Cigna.
Aetna and UnitedHealthcare are deliberate v1 gaps because their public
provider-directory endpoints either require OAuth setup not yet wired
into the analysis pipeline (Aetna) or the URL captured in our
`ProviderDirectoryAPI` table is stale (UHC). See "Out of scope" below.

## Non-goals

- **Not a fraud determination.** A directory match is a triage flag,
  not evidence of billing or active patient assignment. Investigation,
  hearing rights, and reinstatement claims belong to the MCO and the
  excluding agency.
- **Not a per-Medicaid-product breakdown** (v1). Most parent-payer FHIR
  endpoints aggregate commercial + Medicaid managed care directories;
  Anthem's TotalView is the exception with brand-specific URLs (and we
  *do* hit `HealthKeepersInc` directly, which is the Anthem VA Medicaid
  product).
- **Not multi-state** (v1). Virginia first, in service of the 2026-04-23
  CMS SMD letter. PA/OH come after the pattern is proven.
- **Not Aetna or UnitedHealthcare** (v1). Aetna's public Provider
  Directory FHIR endpoint requires OAuth registration not yet wired
  into the analysis pipeline; the UHC URL captured in our
  `ProviderDirectoryAPI` Supabase table is stale (DNS-fails) and the
  current public URL has not been verified. Both are tracked as Stage C
  fast-follows.

## Architecture

```text
high-risk-cohort-export.csv  (already produced by H23)
                │
                ▼
   analysis/h26_mco_exposure_va.py
   • filter: state=VA AND bucket=critical
            AND (oig_excluded OR sam_excluded)
   • for each NPI:
       for each payer endpoint (Anthem HealthKeepers Plus,
       Anthem Blue Cross, Humana, Cigna):
         GET {endpoint}/Practitioner?identifier=<npi>
         classify: matched | not_in_directory | error
         throttle 1 req/sec/host
                │
                ▼
   frontend/public/api/v1/findings/
     mco-exposure-va.json
     mco-exposure-va-detail.json
                │
   ┌────────────┴────────────┐
   ▼                         ▼
 /findings/mco-exposure-va  /states/va  (new MCO-exposure section)
 (loadFinding('mco-exposure-va'))
```

## Data flow

1. **Cohort filter (~milliseconds).** Read the CSV, retain rows where
   `state == "VA"`, `bucket == "critical"`, and `reasons` contains
   `oig_excluded` or `sam_excluded`. Expected size: ~200–400 NPIs.
2. **MCO query loop (~5 minutes).** For each (NPI, MCO) pair, GET
   `{endpoint}/Practitioner?identifier=<npi>` with named UA
   `AINPI-DirectoryQualityBot/1.0 (+https://ainpi.dev/methodology; gene@fhiriq.com)`.
   Throttle: 1 req/sec/host. Single retry on 5xx with 2s jitter, then
   bucket as `error`.
3. **Classification.**
   - HTTP 200 + Bundle with `total ≥ 1` → `matched`
   - HTTP 200 + Bundle with `total == 0` (or missing `total` and
     `entry == []`) → `not_in_directory`
   - Anything else (timeout, 4xx, 5xx, malformed JSON) → `error`
4. **Aggregation.** Per-MCO counts and per-NPI matched-in lists.
5. **Output.** Write the JSON contract + sidecar.
   `notes` and `limitations` are static templates with substituted
   counts; `samples` is the first 10 NPIs with at least one match,
   sorted by `len(matched_in)` desc then `npi` asc for stability.

## Edge cases & risk handling

| Risk | Mitigation |
| --- | --- |
| Confusing `error` with `not_in_directory` | Separate buckets; sidecar reports `errors` per MCO; headline numerator counts only confirmed `matched`. Soft-fail with a warning when any MCO's error rate exceeds 10% of queried NPIs. |
| VA Medicaid coverage gap | Of the 6 VA Medicaid MCO products, only Anthem HealthKeepers Plus is queried as the Medicaid line specifically (via `HealthKeepersInc`). Aetna BH of VA, UHC Community Plan, Sentara Community Plan, Molina Complete Care, and Virginia Premier are all unwired in v1. Lead `limitations` with this; never bury. |
| Aetna OAuth + UHC URL drift | Aetna requires OAuth registration (developerportal.aetna.com); UHC's URL captured in our `ProviderDirectoryAPI` table is stale. Documented as Stage C fast-follows; v1 ships without them. |
| Commercial vs Medicaid line ambiguity | Frame the result as "appears in payer's published provider directory" — never "in Medicaid network specifically." Future work can wire the Medicaid-specific FHIR endpoint per CMS-9115-F. |
| Active flag != network status | Provider directory listing alone does not establish current billing privileges. Frame as data-quality + triage signal aligned with 42 CFR § 455.436. |
| Cohort drift across runs | Stamp `cohort_source: high-risk-cohort-export.csv@<short-sha>` in the output JSON. |
| MCO rate limits / abuse risk | 1 req/sec/host, named UA, single retry on 5xx with jitter. Polite enough that no MCO has paged us so far for the existing `/api/provider-search`. |
| FHIR PractitionerRole filter complexity | Out of scope for v1. We only confirm Practitioner-resource presence; PractitionerRole + Location filtering for Medicaid-line specifics is a follow-up. |

## Testing

**Vitest** (`frontend/tests/`):
- `loadFinding('mco-exposure-va')` returns the typed `ApiV1Finding` shape after the JSON is committed
- Snapshot test of `mco-exposure-va.json` schema for regression

**Python unit tests** (new `analysis/tests/test_h26.py` if pytest infra exists, otherwise inline doctest):
- Bundle parser distinguishes `total=0`, `total≥1`, missing `total`, malformed JSON
- Cohort filter retains only critical+VA+excluded NPIs
- Headline string composer is stable across zero matches, single MCO, all 4 MCOs

**E2E:** skip for v1 — finding page is `force-static`, build catches wiring problems.

## Operations

- **Workflow integration.** Add to `.github/workflows/weekly-refresh.yml`
  immediately after the `high_risk_cohort.py` step so both run in the
  same flight under one `commit_sha`. No new secrets required (MCO FHIR
  APIs are public per CMS-9115-F).
- **Wall-time budget.** ~5 min for ~300 VA critical NPIs × 4 MCOs at
  1 req/sec/host (sequential per host).
- **Failure handling.** If the workflow fails on H26 specifically,
  the existing weekly-refresh "commit if anything changed" step will
  still ship cohort updates; H26 will be regenerated on the next run.

## Questions explicitly out of scope

- Aetna and UnitedHealthcare endpoint wiring (Stage C fast-follow)
- Sentara Community Plan, Molina Complete Care, Virginia Premier wiring
- Per-product Medicaid-line disambiguation for Anthem Blue Cross / Humana / Cigna (which aggregate commercial + Medicaid in their FHIR endpoints)
- Generalization to PA / OH / other states
- Active billing status from T-MSIS claims (Stage 2; separate spec)
- PractitionerRole / Location filtering for Medicaid-line specifics
- Patient-level claim impact

These are deliberate v1 scope cuts to ship the citable headline this
week. Follow-on work tracks each as a separate spec.
