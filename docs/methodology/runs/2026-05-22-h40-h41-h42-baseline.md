The 12-NPI / $3.4 M shortfall in H40 vs H30a is **expected** and explained by CMS's per-(NPI, HCPCS, place-of-service) cell-suppression rule: any cell with fewer than 11 beneficiaries is suppressed in the granular by-Service file, but the suppressed totals are rolled up into the by-Provider aggregate. So 12 NPIs in CY 2023 had *only* low-volume billing activity (e.g., 4-10 beneficiaries per HCPCS) and appear in H30a but not H40.

**Operational implication**: H40's per-claim recoupment cohort is a *lower bound*. H30a's NPI-level totals are the more inclusive count for "did this excluded NPI bill Part B at all in CY 2023." H40's value is the procedure-level detail on the visible (un-suppressed) subset.

### SAM-NPI-join false-positive finding (cohort-builder methodology)

The SAM.gov Public Extract V2 contains an NPI column that is sometimes populated incorrectly: the same NPI appears across records for unrelated individuals, or an NPI is listed against the wrong excluded party due to clerical error at the source. The AINPI cohort builder (`analysis/h25_sam_exclusions.py` + `analysis/build_state_cohort.py`) currently treats any non-empty SAM `npi` field on an active SAM row as a cohort-qualifying signal, without cross-validating against NPPES name.

**Observed false-positive rate among strict-post-exclusion candidates in this run: 3 of 4 (75%).** Sample size is tiny but the pattern is concerning.

**Fix path:** add a NPPES-name-match step. For each cohort row sourced from SAM, fetch the NPI's NPPES name and require at least one token overlap with the SAM-row name. If no overlap, downgrade the row to `bucket=needs-review` rather than `critical`. Publish a corrected `high-risk-cohort-export.csv` and rerun downstream H29 / H30a / H30b / H32 / H40 with the cleaned cohort. Tracked as a follow-up PR; not in scope for this baseline run.

## H42 — Telehealth-dominant post-exclusion billing

**Slug:** `excluded-telehealth-dominant-post-exclusion`
**Script:** `analysis/claims_sources/h42_excluded_telehealth_dominant.py`
**Runtime:** <1 second (pure filter on H40's output).

### Headline — null hypothesis supported

**Zero** federally-excluded NPIs in CY 2023 show ≥80% of post-exclusion Medicare Part B services billed under telehealth-specific HCPCS codes (the publishable threshold). The sensitivity sidecar (60-79%) and high-confidence band (≥95%) are also empty.

### Two competing readings

The null result is consistent with two interpretations and the published headline names both:

1. **Federal exclusion screening for telehealth-specific Part B is working** — the LEIE/SAM screening pipeline is in fact catching telehealth-credentialed excluded providers pre-payment.
2. **The post-exclusion cohort billing Part B is too small** (4 NPIs / 57 service-rows national) for the dominant-share threshold to register at all — denominator pressure, not screening effectiveness.

Both readings are honest. H42 is intentionally framed as a sharpened sub-test, not the headline cohort; H40 (per-claim recoupment unit) is the headline.

## Data quality caveats (load-bearing)

These apply to all three findings:

1. **Cell suppression.** CMS suppresses any (NPI, HCPCS, POS) cell with <11 beneficiaries. Any "X services" count in H40/H42 is a *visible* lower bound; the true billing volume for an excluded NPI may be higher if some of their HCPCS billing was below the threshold. This is also why H40's NPI count (194) is below H30a's (206) — see cross-validation table above.
2. **LEIE-NPI join quality.** OIG LEIE publishes (last name, first name) + NPI for some entries; NPI is missing on many older exclusions and was added retrospectively where CMS could match. The match can be a true exclusion + a different NPI holder sharing the name (false positive), an NPI re-assigned by CMS post-exclusion (rare), or a legitimate match (the desired case). External readers should verify every published candidate against [exclusions.oig.hhs.gov](https://exclusions.oig.hhs.gov/) by NPI lookup before attributing the finding to a specific individual.
3. **SAM.gov scope.** SAM exclusions cover federal procurement and Medicare/Medicaid enrollment debarments. Some SAM-excluded NPIs are excluded only from contracting (not Medicare/Medicaid billing); the AINPI cohort treats all active SAM listings as flags, which is the conservative position but not always the prosecutorial one.
4. **Strict-post-exclusion temporal precision.** The source file publishes year-level totals. The strict-post filter operates at the (NPI, year) grain — an NPI excluded in 2023 then billing in 2023 has same-year activity that we cannot subdivide. The strict-post numerator (4 NPIs) excludes any NPI with exclusion year ≥ 2023.
5. **State attribution.** Per-state CSVs use NPPES practice state (the cohort's state field), which is where the NPI was registered, not necessarily where billing occurred. The CSV row also carries `billing_state` (Rndrng_Prvdr_State_Abrvtn) so cross-state cases are visible per-row.
6. **`estimated_paid_total` is computed, not source-published.** CMS publishes averages per (NPI, HCPCS, POS); the H40 column multiplies the average by the service count. Small rounding drift vs the by-Provider aggregate's exact totals is expected; see cross-validation table.

## Commands (reproducibility)

```bash
# Download source file (one-time, ~3 GB)
curl -L -o frontend/data/cms-claims/partb-by-provider-and-service.csv \
  "https://data.cms.gov/sites/default/files/2025-04/e3f823f8-db5b-4cc7-ba04-e7ae92b99757/MUP_PHY_R25_P05_V20_D23_Prov_Svc.csv"

# H40 — per-claim recoupment unit (~40 sec)
python3 -m analysis.claims_sources.medicare_partb_by_hcpcs

# H42 — telehealth-dominant filter on H40 output (<1 sec)
python3 -m analysis.claims_sources.h42_excluded_telehealth_dominant

# H41 — specialty billing drift (two-pass; needs BQ NPPES access)
python3 -m analysis.h41_specialty_drift
```

All three scripts honor the existing `_cohorts.py` streaming-once partition pattern; per-state CSVs land at `frontend/public/api/v1/states/<state>/h{40,41,42}-*.csv` and the national finding JSON at `frontend/public/api/v1/findings/<slug>.json`.
